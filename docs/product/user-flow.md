# User Flow

## Purpose

이 문서는 방문 수거 서비스에서 사용자가 실제로 이동하는 주요 흐름을 정의한다.
화면 설계, 라우팅 구성, CTA 배치, 예외 처리 설계의 기준으로 사용한다.

## Used By

- Planner
- Designer
- Frontend
- Backend

---

## 1. Flow Principles

- 모든 핵심 흐름은 모바일 우선으로 설계한다.
- 사용자는 가능한 적은 단계로 픽업 요청을 완료할 수 있어야 한다.
- 각 단계에서 다음 행동이 명확해야 한다.
- 상태 확인과 취소는 마이페이지에서 쉽게 접근 가능해야 한다.

---

## 2. Primary User Journeys

MVP에서 우선 정의할 핵심 여정은 아래와 같다.

- 신규 사용자 첫 신청 흐름
- 기존 사용자 재신청 흐름
- 마이페이지에서 요청 상태 확인 흐름
- 요청 취소 흐름
- 알림에서 요청 상세로 진입하는 흐름

---

## 3. New User First Request Flow

### Goal

처음 방문한 사용자가 회원가입 후 픽업 요청과 결제까지 완료한다.

### Flow

1. 랜딩 페이지 진입
2. 서비스 소개 및 핵심 가치 확인
3. `픽업 신청하기` CTA 선택
4. 로그인 필요 안내 확인
5. 회원가입 페이지 이동
6. 회원가입 완료
7. 픽업 요청 페이지 진입
8. 방문 날짜 선택
9. 방문 시간대 선택
10. 주소 및 상세 주소 입력
11. 연락처 확인 또는 입력
12. 수거 품목 카테고리 선택
13. 품목 설명 및 요청사항 입력
14. 입력 내용 검토
15. 결제 페이지 이동
16. 결제 진행
17. 결제 성공
18. 요청 완료 페이지 진입
19. 접수 완료 상태와 다음 안내 확인
20. 마이페이지 또는 요청 상세 페이지로 이동

### Success Condition

- 요청이 생성되고 상태가 `requested`가 된다.

### Failure / Branch Cases

- 회원가입 실패 시 회원가입 단계에 머문다.
- 필수값 누락 시 요청 단계에서 다음으로 진행할 수 없다.
- 결제 실패 시 `pending_payment` 상태 또는 결제 재시도 흐름으로 이동한다.

---

## 4. Existing User Repeat Request Flow

### Goal

기존 사용자가 로그인 후 더 빠르게 다시 픽업을 신청한다.

### Flow

1. 랜딩 페이지 또는 메인 진입
2. 로그인
3. 메인 또는 마이페이지에서 `다시 신청하기` 선택
4. 픽업 요청 페이지 진입
5. 기본 연락처 또는 기본 주소 자동 채움 확인
6. 날짜, 시간대, 품목 정보 입력
7. 결제 진행
8. 요청 완료 페이지 진입
9. 마이페이지에서 새 요청 확인

### Success Condition

- 기존 사용자 정보 일부가 재사용되어 더 짧은 흐름으로 요청 완료

### UX Note

- MVP에서도 최소한 연락처 또는 기본 주소 정도는 재사용 가능하도록 고려할 수 있다.

---

## 5. Request Status Check Flow

### Goal

사용자가 접수 후 현재 진행 상태를 쉽게 확인한다.

### Flow

1. 로그인 상태에서 마이페이지 진입
2. 요청 목록 확인
3. 가장 최근 요청 또는 원하는 요청 선택
4. 요청 상세 페이지 진입
5. 현재 상태 확인
6. 예약 정보, 품목 정보, 메모, 상태 안내 확인

### Key Outputs

- 현재 상태 배지
- 예약 날짜 및 시간
- 주요 요청 정보
- 취소 가능 여부

---

## 6. Cancellation Flow

### Goal

사용자가 취소 가능한 상태에서 직접 요청을 취소한다.

### Entry Points

- 마이페이지 요청 목록
- 요청 상세 페이지

### Flow

1. 사용자가 요청 상세 진입
2. 현재 상태 확인
3. 취소 가능 상태면 `취소하기` 버튼 노출
4. 취소 확인 모달 또는 확인 단계 진입
5. 사용자 최종 확인
6. 취소 처리 완료
7. 상태가 `cancelled`로 변경
8. 취소 완료 안내 표시

### Non-cancellable Branch

1. 사용자가 요청 상세 진입
2. 현재 상태가 `in_progress`, `completed`, `cancelled` 중 하나임
3. 취소 버튼 미노출 또는 비활성화
4. 취소 불가 안내 문구 표시

---

## 7. Notification to Detail Flow

### Goal

사용자가 상태 변경 알림을 보고 바로 관련 요청 상세로 이동한다.

### Flow

1. 사용자가 알림 목록 진입
2. 상태 변경 알림 확인
3. 특정 알림 선택
4. 해당 요청 상세 페이지로 이동
5. 변경된 상태와 상세 정보 확인
6. 필요 시 후속 행동 수행

### Possible Follow-up Actions

- 일정 확인
- 취소 여부 판단
- 재신청

---

## 8. Pending Payment Recovery Flow

### Goal

결제 실패 또는 결제 중단 후 사용자가 다시 결제를 완료한다.

### Flow

1. 사용자가 결제 단계에서 실패 또는 이탈
2. 요청은 `pending_payment` 상태 유지
3. 사용자가 마이페이지 또는 요청 상세에서 미완료 요청 확인
4. `결제 계속하기` 선택
5. 결제 페이지 재진입
6. 결제 재시도
7. 성공 시 `requested` 상태 전환

### Exit Cases

- 사용자가 직접 취소하면 `cancelled`
- 재결제 성공 시 `requested`

---

## 9. Page-Level Flow Map

| From | Action | To |
| --- | --- | --- |
| 랜딩 | 픽업 신청하기 | 로그인 또는 회원가입 |
| 로그인 | 로그인 성공 | 픽업 요청 또는 이전 목적 페이지 |
| 회원가입 | 가입 완료 | 픽업 요청 |
| 픽업 요청 | 입력 완료 후 다음 | 결제 |
| 결제 | 결제 성공 | 요청 완료 |
| 요청 완료 | 내 요청 보기 | 마이페이지 또는 요청 상세 |
| 마이페이지 목록 | 요청 선택 | 요청 상세 |
| 알림 목록 | 알림 선택 | 요청 상세 |
| 요청 상세 | 취소하기 | 취소 확인 |
| 취소 확인 | 취소 확정 | 요청 상세 또는 완료 상태 화면 |

---

## 10. Required UX States

각 주요 흐름에는 아래 상태가 필요하다.

- 로딩 상태
- 빈 상태
- 입력 오류 상태
- 결제 실패 상태
- 취소 불가 상태
- 요청 없음 상태
- 알림 없음 상태

---

## 11. Open Decisions

- 로그인 후 원래 보려던 페이지로 정확히 복귀시킬지
- 요청 완료 후 바로 요청 상세로 보낼지, 완료 전용 페이지를 둘지
- `다시 신청하기`에서 이전 요청 정보를 어느 정도 자동 복사할지
- 결제 대기 요청을 마이페이지 최상단에 강조할지
