# Product Overview

## Purpose

이 문서는 방문 수거 서비스의 전체 목표, 핵심 사용자, 주요 기능, MVP 범위를 정의한다.
이 문서를 기준으로 기획, 디자인, 프론트엔드, 백엔드, 운영 어드민의 방향을 정렬한다.

## Used By

- Planner
- Designer
- Frontend
- Backend
- Admin

---

## 1. Product Summary

### Product Name

문앞수거

### One-line Description

사용자가 의류, 신발, 가방, 소형 생활용품 등을 집 앞에서 편하게 수거 요청할 수 있는 예약형 방문 수거 서비스

### Product Type

예약 기반 방문 수거 서비스

---

## 2. Target Users

### Primary Users

- 1인 가구
- 바쁜 직장인
- 직접 배출 장소까지 이동하기 번거로운 사용자
- 의류, 잡화, 생활용품 등을 한 번에 정리하고 싶은 사용자

### User Characteristics

- 모바일로 빠르게 신청하고 싶어한다.
- 복잡한 배출 절차를 줄이고 싶어한다.
- 수거 진행 상태를 명확하게 확인하고 싶어한다.
- 주소, 시간, 품목 정도만 입력해서 쉽게 완료하길 원한다.

---

## 3. Problem Statement

기존의 물품 정리 및 배출 과정은 사용자 입장에서 번거롭다.

### Problems

- 어떤 품목이 수거 가능한지 바로 이해하기 어렵다.
- 직접 버리러 가거나 배출 규칙을 맞추는 과정이 복잡하다.
- 원하는 날짜와 시간에 맞춰 처리하기 어렵다.
- 요청 이후 현재 진행 상태를 확인하기 어렵다.

---

## 4. Product Goals

### Primary Goals

- 방문 수거 요청 과정을 간단하고 직관적으로 만든다.
- 사용자가 날짜, 시간, 위치, 품목 정보를 쉽게 입력하게 한다.
- 요청 이후 상태를 마이페이지와 알림으로 명확하게 제공한다.

### Portfolio Goals

- 실제 서비스형 제품에 가까운 사용자 흐름을 구현한다.
- 인증, 신청, 결제, 상태 추적, 마이페이지 구조를 포함한다.
- 사용자 앱과 운영 어드민이 함께 돌아가는 구조를 보여준다.

---

## 5. Core Features

### 5.1 User App

- 로그인
- 회원가입
- 비밀번호 찾기
- 로그아웃
- 회원탈퇴

### 5.2 Pickup Request

- 방문 수거 요청
- 날짜 선택
- 시간대 선택
- 주소 및 상세 주소 입력
- 수거 품목 선택
- 추가 요청사항 입력
- 결제 진행
- 요청 완료

### 5.3 My Page

- 요청 내역 조회
- 요청 상세 조회
- 진행 상태 확인
- 취소 가능 여부 확인
- 프로필 관리

### 5.4 Notifications

- 요청 접수 완료
- 결제 완료
- 일정 확정
- 수거 진행 중
- 수거 완료
- 취소 완료

### 5.5 Admin

- 관리자 로그인
- 요청 목록 조회
- 요청 상세 조회
- 요청 상태 변경
- 일정 확정 처리
- 취소 처리
- 운영 메모 확인

---

## 6. Pickup Item Scope

### Supported Categories for MVP

- 의류
- 신발
- 가방
- 소형 생활용품
- 소형 가전
- 기타

### Out of Scope for MVP

- 대형 가구
- 대형 가전
- 위험물
- 즉시 현장 견적이 필요한 품목

---

## 7. MVP Scope

### Included in MVP

- 로그인 / 회원가입
- 방문 수거 요청 플로우
- 결제 UI
- 요청 완료 화면
- 마이페이지 요청 내역 및 상세
- 상태값 표시
- 알림 목록
- 관리자 요청 목록 / 상세 / 상태 변경
- 이용약관 / 개인정보 처리방침 페이지

### Excluded from MVP

- 기사 전용 앱
- 실시간 기사 위치 추적
- 실제 PG 결제 연동
- 실제 문자 발송
- 자동 품목 분류 AI
- 정교한 관리자 통계 대시보드

---

## 8. Product Principles

### Principle 1

신청은 빠르고 단순해야 한다.

### Principle 2

상태는 항상 명확하게 보여야 한다.

### Principle 3

모바일 환경에서 쉽게 완료되어야 한다.

### Principle 4

운영자가 상태를 관리할 수 있어야 서비스가 성립한다.

## 10. Notes for Implementation

- 사용자 서비스는 모바일 우선 UX를 기준으로 설계한다.
- 핵심 흐름은 인증, 예약, 주소 입력, 품목 입력, 결제, 상태 추적이다.
- 운영자 기능은 최소한 요청 상태를 바꿀 수 있을 정도까지 포함한다.
- 상세 정책은 `docs/product/business-rules.md`에서 정의한다.
- 상태값과 상태 전이는 `docs/product/status-model.md`에서 정의한다.
- 화면별 요구사항은 `docs/product/page-requirements.md`에서 정의한다.
- 어드민 요구사항은 `docs/admin/admin-requirements.md`에서 정의한다.
