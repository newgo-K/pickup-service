# Business Rules

## Purpose

이 문서는 방문 수거 서비스의 핵심 운영 규칙을 정의한다.
기획, 화면 설계, API 설계, 데이터 모델링, 관리자 기능 설계 시 공통 기준으로 사용한다.

## Used By

- Planner
- Designer
- Frontend
- Backend
- Admin

---

## 1. Service Scope Rules

### 1.1 Service Type

- 본 서비스는 예약 기반 방문 수거 서비스다.
- 사용자는 원하는 날짜와 시간대를 선택해 수거를 요청한다.
- MVP에서는 실시간 배차가 아닌 예약 접수 흐름을 기준으로 한다.

### 1.2 Supported Pickup Categories

- 의류
- 신발
- 가방
- 소형 생활용품
- 소형 가전
- 기타 운영자가 수거 가능하다고 판단하는 품목

### 1.3 Out of Scope for MVP

- 대형 가구
- 대형 가전
- 위험물
- 즉시 현장 견적이 필요한 품목
- 특수 운송 장비가 필요한 품목

---

## 2. User Rules

### 2.1 Authentication Requirement

- 픽업 요청 생성은 로그인한 사용자만 가능하다.
- 요청 내역 조회, 상세 조회, 취소, 알림 확인은 로그인한 사용자만 가능하다.
- 비로그인 사용자는 서비스 소개와 기본 안내만 볼 수 있다.

### 2.2 User Ownership

- 사용자는 본인이 생성한 픽업 요청만 조회할 수 있다.
- 사용자는 본인의 계정 정보만 수정할 수 있다.
- 한 요청은 하나의 사용자 계정에만 귀속된다.

---

## 3. Pickup Request Rules

### 3.1 Required Fields

- 픽업 요청 생성 시 아래 정보는 필수다.
- 방문 날짜
- 방문 시간대
- 주소
- 상세 주소
- 연락처
- 수거 품목 카테고리
- 수거 품목 설명 또는 메모

### 3.2 Optional Fields

- 공동현관 출입 방법
- 기사 참고사항
- 문 앞에 내놓는 시간 관련 자유 메모
- 사진 첨부 여부는 MVP에서 제외한다.

### 3.3 Request Creation Rule

- 사용자는 한 번의 요청에서 하나의 방문 일정만 생성할 수 있다.
- 한 요청에는 여러 품목을 담을 수 있지만, 하나의 주소와 하나의 시간대만 가진다.
- 요청 생성 후 결제 완료 전까지는 접수 확정으로 보지 않는다.

### 3.4 Address Rule

- 주소는 서비스 가능 지역 내에서만 입력 가능해야 한다.
- MVP에서는 서비스 가능 지역을 고정된 텍스트 목록 또는 선택 목록으로 제한할 수 있다.
- 서비스 가능 지역 밖 주소는 요청 생성이 불가능하다.

### 3.5 Schedule Rule

- 사용자는 운영 가능한 날짜와 시간대만 선택할 수 있다.
- 방문 시간은 운영 슬롯 기준으로 관리한다.
- 사용자가 문 앞에 내놓는 시간은 별도 자유 메모로 처리한다.
- 이미 마감된 시간대는 선택할 수 없다.
- 과거 날짜와 시간은 선택할 수 없다.
- 당일 즉시 수거는 MVP 기본 범위에서 제외한다.

---

## 4. Item Rules

### 4.1 Item Classification

- 사용자는 수거 품목을 카테고리 단위로 선택한다.
- 세부 품목명은 자유 입력 또는 간단한 텍스트 설명으로 보완한다.
- 품목 분류가 애매한 경우 `기타` 카테고리를 허용한다.

### 4.2 Unsupported Item Handling

- 입력 품목이 수거 불가 품목에 해당하면 요청 진행을 막아야 한다.
- MVP에서는 자동 판별보다 안내 문구와 사용자 체크를 우선한다.
- `기타` 카테고리 요청은 자동 허용하고 설명 입력으로 보완한다.

### 4.3 Quantity Rule

- MVP에서는 정밀 수량 산정보다 대략적인 품목 설명을 우선한다.
- 가격은 품목 카테고리 기준 요금 합산 방식으로 계산한다.

---

## 5. Payment Rules

### 5.1 Payment Timing

- 사용자는 픽업 요청 과정에서 결제 단계까지 진행한다.
- 결제가 완료되어야 요청이 최종 접수된다.

### 5.2 MVP Payment Rule

- MVP에서는 실제 PG 연동 없이 결제 UI와 결제 완료 상태만 제공할 수 있다.
- 사용자에게는 카드 결제 UI를 우선 제공한다.
- 개발 및 테스트 환경에서는 모의 결제를 사용할 수 있다.
- 결제 실패 시 요청은 접수 완료 상태로 전환되지 않는다.

### 5.3 Payment Failure Rule

- 결제 실패 시 사용자는 다시 결제를 시도할 수 있다.
- 결제 실패 요청은 일정 시간이 지나면 자동 만료 처리할 수 있다.
- 자동 만료를 구현하지 않더라도 최소한 미완료 요청으로 분리 표시해야 한다.

### 5.4 Pricing Rule

- MVP 가격 정책은 카테고리별 요금 기준으로 설계한다.
- 각 카테고리 금액은 초기 운영값으로 임의 설정할 수 있다.
- 복잡한 견적 엔진은 MVP 범위에서 제외한다.

---

## 6. Status Rules

### 6.1 Core Status Principle

- 모든 요청은 하나의 명확한 현재 상태를 가져야 한다.
- 상태는 사용자에게 이해 가능한 문구로 표시되어야 한다.
- 상태 변경은 정의된 순서 내에서만 가능해야 한다.
- 운영자만 변경할 수 있는 상태 전이가 존재한다.

### 6.2 Minimum Status Set for MVP

- 결제 대기
- 접수 완료
- 수거 예정
- 수거 진행 중
- 수거 완료
- 취소 완료

### 6.3 Status Transition Rule

- 결제 대기에서 결제 완료 시 접수 완료로 변경된다.
- 운영자가 일정을 확정하면 수거 예정으로 변경된다.
- 운영자가 수거 시작 처리하면 수거 진행 중으로 변경된다.
- 운영자가 수거 완료 처리하면 수거 완료로 변경된다.
- 취소 가능 상태에서 사용자 또는 운영자가 취소 처리하면 취소 완료로 변경된다.

### 6.4 Terminal Status Rule

- 수거 완료와 취소 완료는 종료 상태다.
- 종료 상태가 된 요청은 일반 상태로 되돌리지 않는다.

---

## 7. Cancellation Rules

### 7.1 User Cancellation

- 사용자는 취소 가능 상태에서만 직접 취소할 수 있다.
- 기본적으로 `결제 대기`, `접수 완료`, `수거 예정` 상태에서만 취소 가능하게 본다.

### 7.2 Admin Cancellation

- 운영자는 예외 상황에서 취소 가능 상태 요청을 취소 처리할 수 있다.
- 운영 취소 시 취소 사유를 내부적으로 기록할 수 있어야 한다.

### 7.3 Non-cancellable Cases

- `수거 진행 중`, `수거 완료`, `취소 완료` 상태에서는 일반 취소를 허용하지 않는다.
- MVP에서는 상태 기준으로만 취소 가능 여부를 판단한다.

### 7.4 Cancellation Result

- 취소가 완료되면 요청 상태는 `취소 완료`가 된다.
- 결제가 완료된 요청의 환불 처리는 MVP에서 실제 환불 연동 없이 안내 문구로 대체할 수 있다.

---

## 8. Notification Rules

### 8.1 Notification Events

- 요청 접수 완료
- 결제 완료
- 일정 확정
- 수거 진행 시작
- 수거 완료
- 취소 완료

### 8.2 Notification Delivery Rule

- MVP에서는 앱 내 알림 목록을 기본 채널로 사용한다.
- 실제 문자, 이메일, 푸시 발송은 MVP 범위에서 제외할 수 있다.
- 알림은 읽음 여부를 저장할 수 있어야 한다.

### 8.3 Notification Ownership

- 알림은 해당 요청의 소유 사용자에게만 표시된다.
- 사용자는 자신의 알림만 확인할 수 있다.

---

## 9. My Page Rules

### 9.1 Request History

- 사용자는 자신의 요청 목록을 최신순으로 확인할 수 있다.
- 각 목록 항목에는 최소한 요청일, 예약일, 주요 품목, 상태가 표시되어야 한다.

### 9.2 Request Detail

- 사용자는 요청 상세에서 주소, 일정, 품목, 메모, 상태 이력을 확인할 수 있다.
- 취소 가능한 상태라면 상세 화면에서 취소 액션을 제공할 수 있다.

### 9.3 Profile Management

- 사용자는 이름, 연락처, 기본 주소 등 기본 프로필을 수정할 수 있다.
- 민감 정보 변경 시 재인증이 필요하면 추후 확장 범위로 둔다.

---

## 10. Admin Operation Rules

### 10.1 Admin Authentication

- 관리자 기능은 관리자 계정으로 로그인한 사용자만 접근할 수 있다.
- 일반 사용자 계정은 관리자 페이지에 접근할 수 없다.

### 10.2 Admin Visibility

- 운영자는 전체 요청 목록을 조회할 수 있다.
- 운영자는 요청 상세에서 사용자 입력 정보와 현재 상태를 확인할 수 있다.

### 10.3 Admin Actions

- 운영자는 `requested -> scheduled` 상태 전이를 수행할 수 있다.
- 운영자는 `scheduled -> in_progress` 상태 전이를 수행할 수 있다.
- 운영자는 `in_progress -> completed` 상태 전이를 수행할 수 있다.
- 운영자는 취소 가능 상태 요청을 `cancelled`로 변경할 수 있다.

### 10.4 Admin Notes

- 운영 메모는 사용자에게 노출되지 않는 내부 정보로 관리한다.
- 운영 메모 기능은 MVP에서 단순 텍스트 수준이어도 충분하다.

---

## 11. Operational Assumptions for MVP

### 11.1 Admin Scope

- MVP에는 최소 관리자 페이지를 포함한다.
- 관리자 페이지는 요청 목록, 요청 상세, 상태 변경 정도의 최소 기능만 제공해도 된다.

### 11.2 Pricing Assumption

- MVP 가격 정책은 카테고리별 요금 기준으로 운영한다.
- 각 카테고리 금액은 초기 운영값으로 시작한다.

### 11.3 Region Assumption

- MVP 서비스 지역은 제한된 범위에서 시작한다.
- 지역 확장은 추후 운영 데이터에 따라 늘린다.

---

## 12. Validation Rules

- 필수 입력값이 비어 있으면 다음 단계로 진행할 수 없다.
- 잘못된 연락처 형식은 저장할 수 없다.
- 지원하지 않는 날짜 또는 시간은 저장할 수 없다.
- 지원하지 않는 품목은 요청 완료할 수 없다.
- 종료 상태 요청에는 수정 액션을 제공하지 않는다.
- 운영자는 정의된 상태 전이 외의 변경을 수행할 수 없다.

---

## 13. Remaining Open Decisions

- 서비스 가능 지역을 어떤 방식으로 제한할지
- 운영 메모와 취소 사유를 어디까지 구조화할지
- 미결제 요청 자동 만료를 둘지
